## 修复概述
================================================================================

**问题**：EditFileTool在处理多行文本替换时，因换行符格式不匹配导致大量失败
**根因**：文件使用\r\n (Windows)，AICA构造的old_string使用\n (Unix)
**影响**：场景3测试中87%的edit调用失败（13/15次）

**修复方案**：实施场景2中AICA建议的NormalizeLineEndings方法

================================================================================
## 修复内容
================================================================================

### 修改1：添加NormalizeLineEndings辅助方法

**位置**：第218-228行（新增）

```csharp
/// <summary>
/// Normalize line endings to Unix format (\n) to handle cross-platform differences
/// </summary>
private string NormalizeLineEndings(string text)
{
    if (string.IsNullOrEmpty(text))
        return text;

    // Convert all line endings to \n
    return text.Replace("\r\n", "\n").Replace("\r", "\n");
}
```

**功能**：
- 将所有\r\n (Windows) 转换为\n (Unix)
- 将所有\r (Mac Classic) 转换为\n (Unix)
- 统一换行符格式，确保匹配成功

### 修改2：在匹配前归一化内容

**位置**：第115-120行

**修改前**：
```csharp
// Check old_string exists
if (!content.Contains(oldString))
```

**修改后**：
```csharp
// Normalize line endings to handle \r\n vs \n differences
var normalizedContent = NormalizeLineEndings(content);
var normalizedOldString = NormalizeLineEndings(oldString);

// Check old_string exists
if (!normalizedContent.Contains(normalizedOldString))
```

**效果**：
- 文件内容和old_string都转换为统一格式
- Contains检查现在可以正确匹配

### 修改3：归一化唯一性检查

**位置**：第154-155行

**修改前**：
```csharp
var firstIndex = content.IndexOf(oldString);
var lastIndex = content.LastIndexOf(oldString);
```

**修改后**：
```csharp
var firstIndex = normalizedContent.IndexOf(normalizedOldString);
var lastIndex = normalizedContent.LastIndexOf(normalizedOldString);
```

**效果**：
- 唯一性检查使用归一化后的内容
- 避免因换行符差异导致的误判

### 修改4：归一化替换操作

**位置**：第167-169行

**修改前**：
```csharp

    ? content.Replace(oldString, newString)
    : ReplaceFirst(content, oldString, newString);
```

**修改后**：
```csharp
newContent = replaceAll
    ? normalizedContent.Replace(normalizedOldString, newString)
    : ReplaceFirst(normalizedContent, normalizedOldString, newString);
```

**效果**：
- 替换操作使用归一化后的内容
- 确保替换能够成功执行

## 技术细节
================================================================================

### 为什么选择统一为\n而不是\r\n？

1. **简单性**：\n是单字符，处理更简单
2. **通用性**：Unix/Linux/macOS都使用\n
3. **LLM友好**：大多数LLM训练数据使用\n
4. **向后兼容**：Windows系统可以正确处理\n

### 为什么在匹配时归一化，而不是在读取时？

1. **保持原始内容**：文件内容保持原样，不修改用户数据
2. **仅影响匹配**：只在匹配逻辑中归一化，不影响其他操作
3. **最小侵入**：修改范围最小，风险最低
4. **易于回滚**：如果有问题，容易恢复

### 边界情况处理

**空字符串**：
```csharp
if (string.IsNullOrEmpty(text))
    return text;
```
- 避免对null或空字符串进行不必要的处理

**混合换行符**：
```csharp
return text.Replace("\r\n", "\n").Replace("\r", "\n");
```
- 先处理\r\n（Windows）
- 再处理\r（Mac Classic）
- 确保所有格式都被正确转换
