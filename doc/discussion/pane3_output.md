## 1. GitNexus CLI 能力确认

### 1.1 当前 analyze 命令行参数

从 `tools/gitnexus/dist/cli/index.js:19-27` 确认，`analyze` 子命令支持以下选项：

| 参数 | 功能 |
|------|------|
| `-f, --force` | 强制重新索引 |
| `--embeddings` | 启用嵌入生成 |
| `--skills` | 生成社区技能文件 |
| `--skip-git` | 无 .git 也可索引 |
| `-v, --verbose` | 详细日志 |

**结论：不存在 `--no-context-files` 或类似参数。** GitNexus 没有提供跳过 AI 上下文文件生成的选项。

### 1.2 文件生成调用链

```
analyze.js:330  →  generateAIContextFiles(repoPath, ...)
                        ↓
ai-context.js:248-265  →  生成三类文件：
  1. AGENTS.md  (upsertGitNexusSection)
  2. CLAUDE.md  (upsertGitNexusSection)
  3. .claude/skills/gitnexus/  (installSkills，6个skill文件)
```

`generateAIContextFiles()` 在 `analyze.js:330` 被**无条件调用**，没有任何 option 控制。

### 1.3 AICA 对 analyze 的调用方式

`GitNexusProcessManager.cs` 两个方法调用 analyze：

- **`TriggerIndexAsync`** (line 270):
  ```
  {cmd} {analyzeArgs} "{repoRoot}"
  ```
- **`TriggerIndexWithProgressAsync`** (line 357):
  ```
  {cmd} {analyzeArgs} "{repoRoot}"
  ```

其中 `analyzeArgs` = `"{fullPath}" analyze`（bundled）或 `"-y gitnexus@latest analyze"`（npx）。
两者都只传了 `repoRoot` 路径，**没有任何额外 flag**。

## 2. 三种替代方案可行性分析

### 方案 A1: 添加 `--no-context-files` 参数（上游修改）

**可行性: 中 — 需要修改 GitNexus 源码**

- **修改位置**:
  - `tools/gitnexus/dist/cli/index.js:19-27`: 在 analyze 命令添加 `.option('--no-context-files', 'Skip AGENTS.md/CLAUDE.md/.claude/ generation')`
  - `tools/gitnexus/dist/cli/analyze.js:330`: 在 `generateAIContextFiles()` 调用前添加条件判断 `if (!options?.noContextFiles)`
- **AICA 侧修改**:
  - `GitNexusProcessManager.cs` line 111: 将 `analyzeArgs` 从 `$"\"{fullPath}\" analyze"` 改为 `$"\"{fullPath}\" analyze --no-context-files"`
  - NPX fallback (line 38): 同步修改 `NPX_ANALYZE_ARGS`
- **优点**: 根本解决，干净利落
- **缺点**: 需要修改 bundled GitNexus 的 dist 文件（编译产物），或者等上游发版
- **向后兼容**: 如果 GitNexus 升级后不支持此 flag，analyze 命令会报错并退出（Commander.js 对未知 option 默认报错）

**风险评估**: AICA bundled 的是 `dist/` 编译产物（JS文件），直接修改 dist 是可行的但不规范。如果 GitNexus 是 AICA 团队自己维护的上游项目，应在源码层面添加此 option 然后重新构建。

### 方案 A2: 后置清理（analyze 完成后删除文件）

**可行性: 高 — 仅修改 AICA 侧代码**

- **修改位置**: `GitNexusProcessManager.cs`，在 analyze 进程退出后（`TriggerIndexAsync` line 298-299，`TriggerIndexWithProgressAsync` line 384-387）添加清理逻辑
- **具体修改**:
  ```csharp
  // 在 exitCode 获取之后，Debug.WriteLine 之前插入：
  if (exitCode == 0)
  {
      CleanupRedundantFiles(repoRoot);
  }
  ```
  新增私有方法：
  ```csharp
  /// <summary>
  /// Remove AI context files generated by GitNexus analyze that AICA doesn't need.
  /// AGENTS.md content is available via MCP resource gitnexus://setup.
  /// CLAUDE.md and .claude/ are for Claude Code CLI only.
  /// </summary>
  private static void CleanupRedundantFiles(string repoRoot)
  {
      var targets = new[]
      {
          Path.Combine(repoRoot, "AGENTS.md"),
          Path.Combine(repoRoot, "CLAUDE.md"),
      };
      foreach (var file in targets)
      {
          try
          {
              if (File.Exists(file))
              {
                  File.Delete(file);
                  System.Diagnostics.Debug.WriteLine($"[AICA] Cleaned up: {file}");
              }
          }
          catch (Exception ex)
          {
              System.Diagnostics.Debug.WriteLine($"[AICA] Cleanup failed for {file}: {ex.Message}");
          }
      }

      var claudeDir = Path.Combine(repoRoot, ".claude");
      try
      {
          if (Directory.Exists(claudeDir))
          {
              Directory.Delete(claudeDir, recursive: true);
              System.Diagnostics.Debug.WriteLine($"[AICA] Cleaned up: {claudeDir}");
          }
      }
      catch (Exception ex)
      {
          System.Diagnostics.Debug.WriteLine($"[AICA] Cleanup failed for {claudeDir}: {ex.Message}");
      }
  }
  ```
- **需要修改两处调用点**:
  1. `TriggerIndexAsync` — line 298 之后
  2. `TriggerIndexWithProgressAsync` — line 387 之后
- **优点**: 不依赖上游变更，自包含修复
- **缺点**: 文件生成后再删除，存在短暂窗口期；如果用户恰好在 AGENTS.md 中手写了自定义内容（非 GitNexus 生成），会被误删
- **向后兼容**: 无兼容问题，即使 GitNexus 升级后不再生成这些文件，删除不存在的文件不会报错

**误删防护**: `ai-context.js` 使用 `<!-- gitnexus:start -->` / `<!-- gitnexus:end -->` 标记来管理 AGENTS.md 中的 GitNexus 段落。更安全的做法是：
- 若文件只包含 GitNexus 生成的内容 → 整文件删除
- 若文件包含用户自定义内容 + GitNexus 段落 → 仅删除 GitNexus 段落
- 但这增加了复杂度，考虑到 AICA 用户不太会手写 AGENTS.md，简单删除即可

### 方案 A3: .gitignore 注入

**可行性: 低 — 不推荐**

- 不解决文件存在问题（仍会污染用户目录）
- 修改 .gitignore 可能与用户意图冲突
- 如果用户不使用 Git，完全无效

**结论: 排除。**

## 3. 推荐方案

### 3.1 短期方案：A2（后置清理）

**理由**:
- 仅修改 `GitNexusProcessManager.cs` 一个文件
- 新增代码约 30 行
- 不依赖 GitNexus 上游变更
- 失败安全（清理失败不影响索引功能）
- 两个调用点（`TriggerIndexAsync` / `TriggerIndexWithProgressAsync`）修改模式完全一致

### 3.2 长期方案：A1（上游 flag）

在 GitNexus 下次版本更新时添加 `--no-context-files` 参数，届时 AICA 可切换到 A1 并移除 A2 的清理代码。

### 3.3 首次打开 vs 重复打开场景分析

| 场景 | 行为 |
|------|------|
| **首次打开项目** | analyze 运行 → 生成文件 → 清理删除。用户从不看到冗余文件 |
| **重复打开项目** | analyze 检测到 `existingMeta.lastCommit === currentCommit` 时跳过索引（analyze.js:103），不生成文件，无需清理 |
| **force 重索引** | 同首次场景，清理生效 |
| **已有用户自定义 AGENTS.md** | 被 GitNexus 覆写后清理。如果需要保护用户内容，需更复杂的逻辑（见上文误删防护） |

## 5. 与 MCP 内容吸收的关系

### 5.1 AGENTS.md 内容获取不受影响

**关键发现**: `gitnexus://setup` MCP resource 的内容**不是从文件系统读取 AGENTS.md**。

从 `resources.js:385-415` 可以看到，`getSetupResource()` 方法是：
- 从 `backend.listRepos()` 获取已索引仓库列表
- 动态拼接工具表格和资源列表
- 返回的是**从知识图谱数据库动态生成的内容**

而文件系统上的 AGENTS.md 是由 `ai-context.js:generateAIContextFiles()` **单独生成**的，供 Claude Code / Cursor 等工具直接读取文件使用。

**因此**: 删除文件系统上的 AGENTS.md **完全不影响** MCP resource `gitnexus://setup` 的可用性。两者的数据源不同：
- 文件系统 AGENTS.md ← `ai-context.js:generateGitNexusContent()` 生成，内容侧重行为规则
- MCP `gitnexus://setup` ← `resources.js:getSetupResource()` 从数据库动态生成，内容侧重工具参考

这是两套**独立的内容生成路径**，删除文件不会断链。

### 5.2 对 McpRuleAdapter 后续工作的影响

McpRuleAdapter（层面 B）的设计是将 MCP resource 内容解析为 Rule 对象。其数据来源是：
- `gitnexus://setup` MCP resource（通过 `McpClient.ReadResourceAsync`）
- **不是**文件系统上的 AGENTS.md

因此冗余文件清理（层面 A）与 McpRuleAdapter（层面 B）**完全独立**，互不影响：
- A 清理的是文件系统上的 AGENTS.md / CLAUDE.md / .claude/
- B 消费的是 MCP 协议返回的 resource 内容

两者可以独立实施，顺序无关。

### 5.3 内容差异注意事项

需要注意的是，文件系统 AGENTS.md 和 MCP `gitnexus://setup` 的**内容不同**：

| 来源 | 内容特征 | Token 量 |
|------|----------|---------|
| AGENTS.md (文件) | 行为规则（Always/Never/When Debugging/When Refactoring/Self-Check） | ~1800 |
| gitnexus://setup (MCP) | 工具参考表格 + 资源列表 | 不同 |

当前 `AgentExecutor.cs:1033` 读取的是 MCP resource，**不是文件**。因此当前 prompt 中注入的内容来自 `getSetupResource()` 而非 `generateGitNexusContent()`。

如果未来 McpRuleAdapter 需要的是 AGENTS.md 文件中那些行为规则（Always/Never 等），则需要确认这些规则在 MCP resource 中是否可获取。从代码分析看，`getSetupResource()` 返回的是工具参考，**不包含 Always/Never 行为规则**。行为规则仅存在于 `generateGitNexusContent()` 生成的文件版本中。

**建议**: McpRuleAdapter 设计时需明确数据源。如果需要行为规则，可能需要：
1. 在 GitNexus 上游为 MCP 新增一个包含行为规则的 resource（如 `gitnexus://rules`），或
2. 从 bundled 的 `ai-context.js` 中提取规则模板（硬编码在代码中），作为内置 Rule
