# YAML Frontmatter 设计决策总结

## 你的问题

> 为什么设计的时候要使用 YAML + 文本的格式，不直接使用文本格式呢？

## 简短答案

**YAML Frontmatter 提供了元数据层，使规则系统能够实现 3 个关键功能：**

1. **条件激活** - 根据文件路径选择性应用规则
2. **优先级管理** - 解决多个规则的冲突
3. **灵活控制** - 启用/禁用、分类、版本控制

---

## 核心对比

### 纯文本方案的问题

```
.aica-rules/
├── general.md
├── cpp-standards.md
├── python-standards.md
└── security.md
```

**加载方式：** 所有规则都被加载到系统提示中

**问题：**
- ❌ 用户处理 C++ 文件时，Python 规则也被加载
- ❌ 系统提示变得臃肿（包含不相关的规则）
- ❌ LLM 性能下降（需要处理更多信息）
- ❌ 无法设置优先级（规则冲突时无法解决）
- ❌ 无法启用/禁用规则

### YAML + Markdown 方案的优势

```yaml
# cpp-standards.md
---
paths:
  - "**/*.cpp"
  - "**/*.h"
priority: 10
enabled: true
---

# python-standards.md
---
paths:
  - "**/*.py"
priority: 10
enabled: true
---
```

**加载方式：** 根据用户上下文选择性加载规则

**优势：**
- ✅ 用户处理 C++ 文件时，只加载 C++ 规则
- ✅ 系统提示精简（只包含相关规则）
- ✅ LLM 性能更好（专注于相关规则）
- ✅ 可以设置优先级（自动解决冲突）
- ✅ 可以启用/禁用规则

## 3 个关键功能详解

### 1. 条件激活（最重要）

**场景：** 多语言项目

```
项目结构：
├── src/
│   ├── main.cpp
│   ├── utils.h
│   ├── script.py
│   └── app.js
```

**纯文本方案：**
```
用户处理 main.cpp
  ↓
加载所有规则：general.md, cpp-standards.md, python-standards.md, security.md
  ↓
系统提示包含 4 个规则集合
  ↓
LLM 需要处理 Python 和 JavaScript 规则（虽然不相关）
```

**YAML + Markdown 方案：**
```
用户处理 main.cpp
  ↓
提取候选路径：["main.cpp", "src/utils.h", ...]
  ↓
评估规则条件：
  - cpp-standards.md: paths 匹配 ✓ → 激活
  - python-standards.md: paths 不匹配 ✗ → 跳过
  - js-standards.md: paths 不匹配 ✗ → 跳过
  ↓
系统提示只包含相关规则
  ↓
LLM 专注于 C++ 标准
```

**性能对比：**
- 纯文本：系统提示 = 基础 + 4 个规则
- YAML：系统提示 = 基础 + 2 个规则（节省 33%）

### 2. 优先级管理

**场景：** 规则冲突

```yaml
# general.md - 通用规则
---
priority: 5
---
- 函数最多 50 行

# cpp-standards.md - C++ 特定规则
---
paths: ["**/*.cpp"]
priority: 10
---
- 函数最多 100 行（C++ 允许更长）
```

**结果：**
- 通用规则：优先级 5
- C++ 规则：优先级 10（更高）
- 自动选择：C++ 规则的"函数最多 100 行"

**纯文本方案：** ❌ 无法表达优先级，无法自动解决冲突

### 3. 灵活控制

#### 启用/禁用

```yaml
---
enabled: false  # 临时禁用，保留文件
---
```

**优点：** 快速开关，无需删除文件

#### 分类和搜索

```yaml
---
tags: ["backend", "security"]
category: "api-design"
author: "security-team"
---
```

**优点：** 快速找到特定类别的规则

#### 版本控制

```yaml
---
version: "2.0"
changelog: |
  v2.0: 更新了命名规范
  v1.0: 初始版本
---
```

**优点：** 追踪规则演变

## 实际应用示例

### 你的 cpp-standards.md 现在支持的功能

```yaml
---
paths:
  - "**/*.cpp"
  - "**/*.h"
  - "**/*.hpp"
priority: 10
enabled: true
tags: ["cpp", "coding-standards"]
category: "language-specific"
---
```

**这意味着：**

1. **条件激活**
   - 只在处理 C++ 文件时激活
   - 处理 Python 文件时不加载

2. **优先级**
   - 优先级 10（中等）
   - 如果有优先级 20 的安全规则，安全规则优先

3. **启用/禁用**
   - 当前启用（enabled: true）
   - 可以改为 false 临时禁用

4. **分类**
   - 标签：cpp, coding-standards
   - 分类：language-specific
   - 便于搜索和组织

## 为什么这个设计很重要

### 问题 1：系统提示膨胀

**纯文本方案：** 所有规则都被加载
```
系统提示大小 = 基础提示 + 所有规则
            = 5000 + (500 × 5)
            = 7500 字符
```

**YAML 方案：** 只加载相关规则
```
系统提示大小 = 基础提示 + 相关规则（平均 2 个）
            = 5000 + (500 × 2)
            = 6000 字符
```

**节省：** 20% 的系统提示大小 → 更快的 LLM 响应

### 问题 2：规则冲突

**纯文本方案：** 无法解决
- 两个规则说不同的东西
- LLM 不知道哪个优先

**YAML 方案：** 自动解决
- 优先级高的规则优先
- 系统自动选择

### 问题 3：规则管理

**纯文本方案：** 困难
- 想禁用规则？删除文件
- 想找特定类别的规则？手动搜索
- 想追踪规则版本？无法实现

**YAML 方案：** 简单
- 想禁用规则？改 enabled: false
- 想找特定类别的规则？搜索 tags
- 想追踪规则版本？查看 version 字段

这个设计来自 **Cline 项目**（一个成熟的 AI 编程助手），已被证明是有效的。

**Cline 的规则系统：**
- 支持条件激活
- 支持优先级管理
- 支持启用/禁用
- 支持分类和搜索
- 支持版本控制

**我们采用了相同的设计，因为它已经在实战中验证过。**
