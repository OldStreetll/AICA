## 二、优化方案

### Phase 1: System Prompt 精化 (高 ROI, 低风险)

**目标**: 通过精化 system prompt 改善 7 个可优化问题

**文件**: `src/AICA.Core/Prompt/SystemPromptBuilder.cs`

#### 1.1 正文与 completion 一致性规则

**解决**: P1-009 (方法计数自相矛盾)

```
新增规则:
- 正文中出现的数字必须与 attempt_completion 中的数字完全一致
- 如果正文说"共 48 个方法"，completion 也必须说"48 个"，不得四舍五入或重新计算
- 如果不确定数量，使用"约 N 个"或"N+ 个"而非给出不同的具体数字
```

#### 1.2 condense 后行为指导

**解决**: P1-012 (condense 后未回答原始问题), P1-013 (摘要未保留文件操作历史)

```
新增规则:
- condense 摘要必须包含: (1) 读取/修改过的文件列表 (2) 关键发现 (3) 当前任务状态
- condense 完成后，必须继续处理用户的最新请求，不得开始新任务
- 如果 condense 导致信息丢失，应主动告知用户并建议重新查询
```

#### 1.3 复杂任务输出格式指导

**解决**: P1-016 (架构概览过于简略)

```
新增规则:
- 当用户请求"完整概览"、"全面分析"、"架构概览"时:
  - 必须调用相关工具获取信息（不得仅依赖已有上下文）
  - 输出必须包含: 项目列表、分层关系、技术栈、关键依赖
  - attempt_completion 摘要不少于 10 行
```

#### 1.4 工具替换告知规则

**解决**: P2-003 (用 list_dir 替代 dir 但未告知)

```
新增规则:
- 当使用与用户指定不同的工具时，必须在回复开头说明:
  "注意: 使用了 {实际工具} 替代 {用户指定的工具}，因为 {原因}"
```

#### 1.5 搜索范围规则

**解决**: P2-005 (未主动补充探索), P2-006 (搜索范围不完整)

```
新增规则:
- 当用户说"所有"、"全部"、"整个项目"时，搜索范围必须是整个 workspace，不得局限于单个子目录
- 对于分析类任务，如果当前上下文不足以给出完整答案，必须主动调用工具补充信息
```

**预期影响**: 改善 7 个问题，通过率从 76% 提升至 ~85%

### Phase 2: condense 摘要质量增强 (中 ROI, 低风险)

**目标**: 改善 condense 后的上下文保留质量

**文件**: `src/AICA.Core/Agent/AgentExecutor.cs` — `BuildAutoCondenseSummary`

#### 2.1 增强自动摘要生成

当前 `BuildAutoCondenseSummary` 提取的信息不够结构化。优化为：

```
摘要模板:
## 文件操作记录
- 读取: file1.h, file2.cpp, ...
- 创建: test_note.txt
- 修改: test_note.txt

## 关键发现
- CalcEngine.h 定义了 CCalcEngine 类，包含 22 个公共方法
- Command.h 定义了 4 个枚举类型

## 当前任务状态
- 用户最后的请求: "我之前读取了哪些文件？"
- 当前进度: 已完成 12 轮对话
```

#### 2.2 condense 后注入上下文提示

在 condense 完成后，向 LLM 注入一条提示消息：

```csharp
condensed.Add(ChatMessage.System(
    "[Post-condense instruction] The conversation was condensed. " +
    "Answer the user's LATEST message based on the summary above. " +
    "Do NOT start a new task or explore new files unless the user asks."));
```

**预期影响**: TC-A14 从 PARTIAL 提升至 PASS

### Phase 3: LLM 行为问题的间接缓解 (低 ROI, 需持续迭代)

这些问题无法通过代码直接修复，但可以通过以下策略间接缓解：

#### 3.1 摘要信息丢失缓解

**问题**: P1-001/002/003/004/010 (LLM 在摘要中丢失细节)

**策略**: 在 `attempt_completion` 的工具描述中增加结构化模板要求

```
attempt_completion 工具描述优化:
- result 参数描述改为: "结构化摘要，必须包含以下部分（如适用）：
  1. [文件结构] 列出所有类、枚举、接口名称（完整名称含前缀/后缀）
  2. [方法清单] 列出所有 public 方法，分类展示
  3. [依赖关系] #include 列表和关键依赖
  4. [数量统计] 精确数字（类数、方法数、文件数等）
  5. [关键发现] 值得注意的设计模式、异常、特殊结构"
```

#### 3.2 过度推断/虚构缓解

**问题**: P1-014/015 (虚构设计模式、无依据的功能声称)

**策略**: 在 system prompt 中增加"证据要求"规则

```
新增规则:
- 分析设计模式时，每个模式必须附带具体的代码证据（文件名+类名+方法名）
- 如果无法提供代码证据，则不得声称该模式存在
- 禁止使用"可能"、"推测"等模糊词汇来声称功能存在
- 只报告你在代码中实际看到的内容，不要推断未读取的代码可能包含什么
```

#### 3.3 数量精确性缓解

**问题**: P1-009 (计数不一致), P2-002 (未提及行数)

**策略**: 要求 LLM 在计数时使用工具验证

```
新增规则:
- 当需要报告精确数量（方法数、文件数、行数等）时:
  - 优先使用 grep_search 的统计结果（max_results 设置足够大）
  - 不要手动数数或估算
  - 如果工具返回了截断结果，使用工具报告的总数而非手动计数
```

**预期影响**: 减少约 30-50% 的 LLM 行为问题出现频率

## 五、不建议优化的问题

以下问题属于 LLM 固有局限，投入产出比低，建议接受现状：

1. **P1-001/002 (摘要遗漏结构信息)**: read_file 返回完整文件内容，LLM 在总结时必然丢失部分信息。可通过 Phase 3.1 的结构化模板部分缓解，但无法完全解决
2. **P2-001 (读取文件只给摘要)**: 这实际上是一个设计权衡 — 展示完整文件内容会占用大量 UI 空间。AICA 的 UI 层已经在 ToolLogsHtml 中展示了完整内容，completion 摘要给出要点即可
3. **P2-004 (遗漏 Factory 模式)**: 设计模式识别是主观性较强的分析，不同工程师对同一代码可能有不同解读。LLM 识别了主要模式（Observer/Visitor/Command）已属良好
