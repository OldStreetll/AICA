# AICA 产品设计研讨会纪要

> 版本: v1.0 | 日期: 2026-03-22
> 参会角色: AI Agent 产品经理 [PM] / AI Agent 开发工程师 [ENG] / 单位领导 [LD]
> 定位: 三角色产品设计讨论 → 定位、功能、痛点、技术路线的收敛决策

---

## 一、背景与输入

### 1.1 会议触发

现有路线图（AICA_ProjectReview_and_Roadmap.md v1.1）存在以下不足：

1. 按技术模块划分步骤，缺乏用户可感知的里程碑
2. 三类目标用户（研发/QT/算法）被当作铁板一块，未做差异化
3. AICA 与 OpenCode 的功能边界模糊
4. 跨文件实现任务（F4）的失败处理缺失——用户看到的是"Agent Error 已达到最大迭代次数"
5. 评分体系和预估数据缺乏实测验证
6. 时间线零缓冲，单人开发不现实

### 1.2 关键输入数据

**用户规模与角色分布：**

| 角色 | 人数 | 技术栈 | 日常工作 |
|------|------|--------|---------|
| 研发工程师 | ~20 | C/C++, MISRA C | 运动控制、PLC 逻辑、轴控代码 |
| QT 界面工程师 | ~20 | Qt-C++, QSS | HMI 界面、参数配置页、状态面板 |
| 算法工程师 | ~20 | C/C++, 数学库 | 插补算法、路径规划、曲面拟合 |
| 架构/系统工程师 | ~10 | 全栈 C/C++ | 数控系统架构级重构（→ 使用 OpenCode） |

**试用反馈关键案例：**

用户指令：
> "struct MinCurve 本来是用来规划单个型腔刀路时，保存最内部轮廓的结构体。在 Area.h 和 Area.cpp 中实现接口，Area 能适应的最小刀具直径；最后用这个数据来和刀具表比较，获取刀具表中最适合的刀具号（这个不实现在 Area 中，实现在 procparaminterface 中）"

结果：Agent 执行了 48 次工具调用，撞上 50 次迭代上限，报 Agent Error，无任何有效产出。

**考核指标：** 任务完成率 + 代码产出量（实测数据，非估算）

## 二、AICA 定位（结论 1）

### 2.1 核心定位

> **AICA 是 VS2022 内嵌的 C/C++ 日常编码助手，面向研发工程师、QT 界面工程师和算法工程师。核心价值是"在 VS 里随手就能用，理解你的代码和规范"。**

不是通用 AI Agent，不是代码生成器，不是 OpenCode 的替代品。

### 2.2 AICA 与 OpenCode 的分工

| 维度 | AICA | OpenCode |
|------|------|----------|
| 使用者 | 60 人：研发/QT/算法 | 10 人：架构/系统 |
| 使用场景 | VS2022 里随手用，改 1-5 个文件 | 终端里做计划性重构，改 10-50+ 个文件 |
| 交互方式 | 选中代码右键、聊天窗口提问 | 终端输入详细指令 |
| 使用频率 | 每天多次 | 每周 1-2 次 |
| 核心要求 | **快、准、不出错** | **深、广、能收尾** |

### 2.3 定位理由

1. 涉密环境中 AICA 是**唯一的 VS 内嵌 AI 工具**，VS 集成是不可替代的优势
2. 60 名日常用户的高频需求（理解代码、生成代码、修 Bug）不需要重型 Agent 能力
3. OpenCode 已经覆盖了大规模重构场景，AICA 不需要重复建设
4. MiniMax-M2.5 的能力天花板决定了 AICA 不适合做复杂 Agent 任务

## 三、三类用户画像与需求分析

### 3.1 研发工程师（20 人）— 运动控制/PLC 逻辑

**工作特征：**
- 代码严肃：运动控制代码一个 bug 可能导致机床撞机，代价极高
- 遗留代码多：很多模块是 10-20 年前写的 C 代码，注释少或注释过时
- 强规范约束：MISRA C 子集、公司内部安全编码规范
- 硬件相关：代码和硬件行为强耦合，AI 难以理解硬件语义

**需求优先级：**

| 优先级 | 功能 | 具体场景 | AICA 可行性 |
|--------|------|---------|------------|
| P0 | 遗留代码解释 | "这段轴控状态机在干什么？每个 case 分支的含义？" | 高 |
| P0 | 规范合规生成 | 生成的代码自动遵循 MISRA C + 公司规范 | 高 |
| P1 | 错误模式分析 | "这段 PLC 逻辑在什么条件下会进入死循环？" | 中 |
| P1 | 跨文件实现 | "在 CAxis.h/cpp 里加一个新的运动模式" | 中（50% 完成率） |
| P2 | 代码审查辅助 | "检查这段代码有没有违反 MISRA Rule 11.3" | 中 |

**限制说明：** AICA 不理解硬件语义。"CAxis::SetPosition 在什么条件下会导致跟踪误差过大"这类问题需要伺服驱动器的物理知识，超出 AI 能力边界。

### 3.2 QT 界面工程师（20 人）— HMI 界面

**工作特征：**
- 大量重复性工作：设置界面、参数配置页面、状态显示面板
- 模式化：signal/slot 绑定、.ui 文件加载、.qss 样式应用
- 频繁修改：UI 需求变更频繁，经常改布局和交互逻辑

**需求优先级：**

| 优先级 | 功能 | 具体场景 | AICA 可行性 |
|--------|------|---------|------------|
| P0 | QT 代码模板生成 | "生成一个参数设置对话框，包含 3 个 SpinBox 和确认取消按钮" | 高 |
| P0 | Signal/Slot 查找 | "QPushButton clicked 信号连接到了哪个 slot？" | 高（GitNexus） |
| P1 | .qss 样式生成 | "生成一个暗色主题的按钮样式" | 高 |
| P1 | UI 逻辑解释 | "这个对话框的数据流是怎样的？" | 高 |
| P2 | 界面代码重构 | "把这个 QWidget 拆成 MVC 结构" | 中 |

**关键判断：QT 工程师是 AICA 最容易出效果的群体。** 原因：
1. QT 代码模式化程度高，LLM 生成质量相对可靠
2. 单文件居多（一个对话框 = 一个 .h + 一个 .cpp），不需要复杂跨文件
3. Signal/Slot 查找正好是 GitNexus `context()` 工具能解决的
4. .qss 生成是纯文本模板，对模型能力要求低

**决策：QT 工程师作为 AICA 的第一批重点上线用户。**

### 3.3 算法工程师（20 人）— 插补/路径规划

**工作特征：**
- 数学密集：代码里大量线性代数、微积分、数值方法
- 精度敏感：算法的数值稳定性、边界条件处理非常关键
- 论文导向：经常需要把论文里的算法实现为 C++ 代码
- 独立模块：算法模块通常相对独立，接口清晰

**需求优先级：**

| 优先级 | 功能 | 具体场景 | AICA 可行性 |
|--------|------|---------|------------|
| P0 | 算法代码解释 | "这段 NURBS 插补算法的参数化逻辑是什么？" | 高 |
| P0 | 辅助实现 | "实现一个 Runge-Kutta 4 阶积分器，精度 1e-6" | 中 |
| P1 | 数值稳定性检查 | "这段矩阵求逆在什么条件下会数值不稳定？" | 中 |
| P1 | 边界条件补全 | "这个插补函数没有处理首末段加减速的情况" | 中 |
| P2 | 算法优化 | "这段 FFT 实现有没有向量化优化空间？" | 低 |

**关键判断：** 算法工程师的需求重心是 **理解 > 辅助实现 > 独立实现**。MiniMax-M2.5 的数学推理能力有限，不应让 AICA 独立编写复杂算法，而是帮工程师理解现有算法代码、找到 bug、补全边界条件。

### 3.4 不覆盖的用户（→ OpenCode）

| 角色 | 人数 | 使用工具 | 原因 |
|------|------|---------|------|
| 架构工程师 | ~5 | OpenCode | 大规模重构需要并行子任务 + clangd LSP |
| 系统工程师 | ~5 | OpenCode | 跨模块 API 迁移需要 LSP 级语义理解 |

## 四、AICA 功能清单（结论 2）

### 4.1 必须实现的功能

| 功能 | 描述 | 面向角色 | 里程碑 | 技术可行性 |
|------|------|---------|--------|-----------|
| **F1 代码理解** | 右键"解释代码"含调用链、执行流、模块归属 | 全部 60 人 | M1 | 高（GitNexus context()） |
| **F2 规范生成** | 生成代码自动遵循 Allman/m_/MISRA/Qt 规范 | 全部 60 人 | M1 | 高（规则注入） |
| **F3 Bug 定位** | 分析错误信息 + 定位相关代码 + 建议修复 | 全部 60 人 | M1 | 中（依赖模型推理） |
| **F4 跨文件实现** | 1-5 文件的功能实现，含进度汇报和优雅中断 | 研发 + 算法 | M2 | 中（50% 一次完成率） |
| **F5 QT 模板** | QT 对话框/Widget 代码生成 + signal/slot 查找 | QT 20 人 | M2 | 高 |
| **F6 算法辅助** | 算法代码解释 + 边界条件检查 + 辅助实现 | 算法 20 人 | M3 | 中 |
| **F7 跨会话记忆** | 记住项目背景、构建系统、依赖关系 | 全部 60 人 | M2 | 高（Memory Bank） |
| **F8 修改影响预警** | 编辑前展示受影响的代码和执行流 | 研发 + 算法 | M2 | 高（GitNexus impact()） |

### 4.2 明确不做的功能

| 功能 | 原因 |
|------|------|
| 10+ 文件大规模重构 | → OpenCode，需要并行子任务 + LSP |
| 硬件语义理解 | 超出 AI 能力边界 |
| 独立编写复杂算法 | MiniMax-M2.5 数学推理不可靠 |
| 代码编译和运行 | VS2022 自身功能 |
| 跨模块 API 迁移 | → OpenCode，需要 LSP 级语义理解 |

### 4.3 F4 跨文件实现——特殊设计

F4 是用户日常高频需求，但 MiniMax-M2.5 的能力上限决定了一次完成率约 50%。产品设计的关键转变：

> **F4 的成功标准不是"一次完成"，而是"每次都有进展、多次能完成"。**

具体设计：

| 机制 | 说明 | 用户感知 |
|------|------|---------|
| 任务分解引导 | Complex 任务强制拆分子步骤后执行 | 用户看到"步骤 1/4 已完成" |
| 进度可见 | Plan 每完成一步在 UI 上标记 | 用户知道 Agent 正在做什么 |
| 优雅中断 | 接近迭代上限时汇报已完成工作 | 不再是"Agent Error"而是"已完成步骤 1-2，步骤 3 未完成" |
| 已完成保留 | 成功编辑的文件保留，不回滚 | 即使中断，之前的修改仍然有效 |
| 断点续做 | 用户发送"继续"从上次进度接着做 | 分 2-3 次对话完成复杂任务 |

**F4 完成率预测（基于 MiniMax-M2.5 能力评估）：**

| 场景 | 当前 | 优化后 | 含断点续做 |
|------|------|--------|-----------|
| 2 文件、明确指令 | ~50% | ~80% | ~90% |
| 3-5 文件、半模糊指令 | ~20% | ~50% | ~70% |
| 5+ 文件、复杂逻辑 | ~5% | ~25% | ~40% |

## 五、痛点优先级（结论 3）

### 5.1 P0 痛点——不解决就没人用

| # | 痛点 | 影响人群 | 当前状态 | 解决方案 |
|---|------|---------|---------|---------|
| P0-1 | 代码解释缺乏上下文——不知道函数在系统中的角色 | 全部 60 人 | 只能解释选中片段 | GitNexus context() 提供调用链 + 执行流 + 聚类 |
| P0-2 | 生成的代码不遵循公司规范——花括号、命名、注释格式全不对 | 研发 + QT | 通用 prompt | C/C++ 规范注入 + Qt 规则 + 语言检测 |
| P0-3 | 跨文件任务撞上迭代上限——48 轮白跑，没有任何产出 | 研发 + 算法 | 直接报 Agent Error | 优雅中断 + 进度汇报 + 断点续做 |
| P0-4 | 编辑文件频繁失败——模型生成的代码缩进不对就报错 | 全部 60 人 | 精确匹配 | 3 层 fallback 容错 |

### 5.2 P1 痛点——解决后体验显著提升

| # | 痛点 | 影响人群 | 解决方案 |
|---|------|---------|---------|
| P1-1 | 每次新会话都要重新解释项目背景 | 全部 60 人 | Memory Bank |
| P1-2 | QT 工程师找不到 signal/slot 连接关系 | QT 20 人 | GitNexus context() + Qt signal/slot 解析 |
| P1-3 | 长对话后 Agent 忘了自己在做什么 | 做复杂任务的人 | 结构化 Condense（程序化提取 + 保护字段） |
| P1-4 | 不知道修改会影响哪些地方 | 研发 + 算法 | GitNexus impact() 爆炸半径分析 |

### 5.3 P2 痛点——锦上添花

| # | 痛点 | 影响人群 | 解决方案 |
|---|------|---------|---------|
| P2-1 | 测试生成框架硬编码为 xUnit——应该是 Google Test 或 CppUnit | 研发 + 算法 | 语言检测 + 测试框架配置 |
| P2-2 | 算法代码的数值稳定性分析 | 算法 20 人 | 算法场景 prompt（中期） |
| P2-3 | 并发排队体验差（超时报错而非排队等待） | 规模化后 | 请求队列（步骤 8） |

## 六、技术路线（结论 4）

### 6.1 里程碑式路线图

以**用户可感知的效果**驱动，取代原来按技术模块划分的 9 步骤方案。

```
2026年3月底          4月中旬            5月中旬             7-8月
    │                 │                 │                  │
    ▼                 ▼                 ▼                  ▼
[基础修复]        [里程碑 M1]       [里程碑 M2]        [里程碑 M3]
 SEC-01            F1 代码理解可靠    F2+F4 可用          全角色可用
 BF-02/06          F2 规范生成        F7 跨会话记忆       F6 算法辅助
 基础稳定           GitNexus 集成      F8 影响预警         规模化
                   F3 Bug 定位        Edit 容错           请求队列
                                     优雅中断             使用监控
                   → QT工程师先用     → 研发工程师开始用    → 算法工程师开始用
```

### 6.2 里程碑 M1：F1 可靠（代码理解做到 85%+）

**目标：** 让 60 个人都愿意尝试 AICA 的门槛。QT 工程师先上线。
**交付时间：** 4 月中旬
**前置工作（3 月底前）：** 基础修复（SEC-01, BF-02, BF-06）

| 技术任务 | 对应原步骤 | 内容 | 预估工时 |
|---------|-----------|------|---------|
| 修 SEC-01、BF-02、BF-06 | 步骤 1 | 安全去重 + 推理泄漏 + 复杂度误分类 | 2-3 天 |
| C/C++ 规范注入 | 步骤 2 | .aica-rules/ 下 5 个规范文件 | 1 天 |
| 语言检测 + 场景 prompt | 步骤 3 部分 | ProjectLanguageDetector + 右键命令语言感知 | 2 天 |
| GitNexus MCP 集成 | 步骤 3.5 | MCP Client + 进程管理 + 工具桥接 | 3-5 天 |
| 工具调用可靠性验证 | 新增 | 用 10 个真实任务验证 MiniMax-M2.5 工具调用准确率 | 1 天 |
| **缓冲** | | 意外问题、调试时间 | **3-5 天** |

**M1 用户可感知的效果：**
- 右键"解释代码"包含调用链、所属模块、上下游关系
- QT 工程师问"这个 slot 被谁连接了"能得到精确答案
- 生成的代码自动使用 Allman 花括号、m_ 前缀
- 代码解释不再触发不必要的 Complex 规划流程

**M1 验收标准：**

| 指标 | 目标 | 衡量方式 |
|------|------|---------|
| F1 代码理解成功率 | ≥80% | 抽样 20 个理解任务，人工判断 |
| F2 规范合规率 | ≥70% | 生成 10 段 C++ 代码，检查规范遵循 |
| GitNexus 索引可用 | C++ 项目索引 <30 秒 | 用公司项目实测 |
| 日活用户 | 10-15 人 | 使用日志 |

**M1 完成后操作：** 部署给 20 名 QT 工程师 + 部分研发工程师，收集早期反馈。

### 6.3 里程碑 M2：F2+F4 基本可用

**目标：** 代码生成遵循规范 + 跨文件实现有进展（不要求一次完成，但每次都有产出）。研发工程师开始正式使用。
**交付时间：** 5 月中旬

| 技术任务 | 对应原步骤 | 内容 | 预估工时 |
|---------|-----------|------|---------|
| EditFileTool 3 层容错 | 步骤 3 | 缩进忽略、空白规范化、相似片段提示 | 3 天 |
| 任务分解 prompt 引导 | 新增 | Complex 任务强制拆分子步骤后执行 | 2 天 |
| 优雅中断 + 进度汇报 | 新增 | 迭代上限前保存进度、汇报已完成工作 | 2 天 |
| 断点续做 | 新增 | 新会话可读取上次进度，发送"继续" | 2 天 |
| Memory Bank | 步骤 4 | 跨会话记住项目背景 | 3 天 |
| Condense 结构化 | 步骤 5 | 程序化提取已读/已改文件 + 保护字段 | 3 天 |
| M1 反馈修复 | 新增 | QT 工程师试用后的问题修复 | 3-5 天 |
| **缓冲** | | | **3-5 天** |

**M2 用户可感知的效果：**
- 生成的 C++ 代码自动遵循 Allman 花括号、m_ 前缀、MISRA 子集
- 跨文件实现任务即使没有一次做完，用户看到进度、知道 Agent 做了什么
- 失败时已完成的编辑保留，用户可发送"继续"接着做
- 新会话不需要重新解释项目背景
- 编辑文件因缩进差异失败的概率大幅降低

**M2 验收标准：**

| 指标 | 目标 | 衡量方式 |
|------|------|---------|
| F1 代码理解成功率 | ≥85% | 抽样 20 个任务 |
| F2 规范合规率 | ≥85% | 生成 10 段代码检查 |
| F4 一次完成率（2 文件） | ≥70% | 抽样 10 个任务 |
| F4 含断点续做完成率 | ≥60% | 同上 |
| Edit 成功率 | ≥90% | 统计 edit 工具调用成功/失败比 |
| 日活用户 | 25-35 人 | 使用日志 |

### 6.4 里程碑 M3：全角色可用 + 规模化

**目标：** 算法工程师场景覆盖 + 支撑 50-60 人同时使用。
**交付时间：** 7-8 月

| 技术任务 | 内容 | 预估工时 |
|---------|------|---------|
| 算法场景 prompt 优化 | 数学解释、数值分析、边界条件检查的专用 prompt | 2 天 |
| QT 专用增强 | .ui 文件关联、QObject 层次分析 | 3 天 |
| 请求队列 + 排队提示 | 并发超限时排队而非超时 | 1 周 |
| 使用监控 | token 用量、成功率、活跃用户数看板 | 1 周 |
| 测试框架自适应 | 语言检测后选择 Google Test / CppUnit | 1 天 |
| M2 反馈修复 | 研发工程师试用后的问题修复 | 1 周 |

**M3 验收标准：**

## 附录：MinCurve 失败案例根因分析

**用户指令：**
> struct MinCurve 本来是用来规划单个型腔刀路时，保存最内部轮廓的结构体。在 Area.h 和 Area.cpp 中实现接口，Area 能适应的最小刀具直径；最后用这个数据来和刀具表比较，获取刀具表中最适合的刀具号（这个不实现在 Area 中，实现在 procparaminterface 中）

**失败表现：** 执行 48 次工具调用，撞上 50 次迭代上限，报 Agent Error。

**根因拆解：**

| 根因 | 消耗的无效迭代 | 对应修复 |
|------|-------------|---------|
| 模型未做任务分解，一口气从头干到尾 | ~10 轮（重复搜索同一内容） | 任务分解 prompt 引导 |
| EditFileTool 精确匹配失败后重试 | ~8 轮（缩进差异导致反复尝试） | 3 层 fallback 容错 |
| 第 25 轮后模型忘了要做什么（condense 信息丢失） | ~10 轮（重新搜索已读过的文件） | 结构化 Condense |
| 模型在 Area.h 和 Area.cpp 之间反复切换 | ~5 轮 | 跨文件重构 prompt 引导 |
| 到 procparaminterface 实现时上下文已满 | ~15 轮（上下文裁剪后迷失方向） | 动态迭代上限 + 优雅中断 |

**优化后预期：** 有效迭代降至 25-30 轮，配合迭代上限提升至 80，预计 50-60% 的概率可一次完成。即使未完成，优雅中断保证 Area.h/Area.cpp 的修改已保留，用户可发送"继续完成 procparaminterface 部分"。
