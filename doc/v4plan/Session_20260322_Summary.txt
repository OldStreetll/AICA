阶段 4：产品战略定位（核心转折点）
================================================================================

【关键转折】
开发者拒绝了"修修补补"的战术路线:
"漏洞的填补是暂时的，没有从架构层面和产品层面系统性的、
 全局的考虑AICA的产品方向。"

收集到的关键信息:
- 公司: 武汉华中数控，代码涉密，无法用Cursor/Claude Code
- 用户: ~80名C/C++开发者，6人正在试用旧版
- LLM: MiniMax-M2.5（220B参数，私有部署，多模态）
- 开发团队: 仅1人，全程vibe coding
- 技术栈: Qt-C++, MISRA C标准, VS2022
- 交付节点: 2026-03-31

并发性能数据:
  20并发 + 10K input → TTFT 23秒
  50并发 + 10K input → TTFT 102秒
  50并发 + 20K input → TTFT 207秒

试用反馈5个痛点:
  痛点1 无输出反馈    → 已修复
  痛点2 输出不稳定    → 极偶发
  痛点3 网络不稳定    → 极偶发
  痛点4 复杂指令处理  → 仍显著（唯一剩余的主要问题）
  痛点5 超出循环上限  → 已修复（MaxAgentIterations=50）

三面结构性墙壁（AICA vs Cursor）:
  模型能力: MiniMax-M2.5 vs Claude/GPT-4o
  平台:     VS2022 VSIX + IE Trident vs Electron
  并发:     私有部署物理限制 vs 云端弹性扩容

战略重新定位:
  FROM: "企业版Cursor"（追求通用AI agent能力）
  TO:   "企业内部C/C++ AI编程助手"（在垂直领域做到极致）

四阶段路线图:
  第1阶段: 稳定交付（2026-03）
  第2阶段: C/C++专家化（2026-04~05）
  第3阶段: 规模化到80人（2026-06~08）
  第4阶段: 架构升级（2026-09+）

核心洞察:
"对于涉密环境中的唯一AI工具，竞争力不来自追赶通用AI agent的
 全面能力，而来自在特定领域的不可替代性。"

我假设Qwen3-Coder-Next比MiniMax-M2.5强，被开发者质疑——
性能测试只测了吞吐量/延迟，不代表输出质量。

用户确认:
- "C/C++专家化方向完全符合公司需求，极其认同"

- 上级希望AICA持续迭代进化

V4路线图深度讨论
================================================================================

已完成工作中的5个隐患:

  1. SEC-01: R5 dedup不区分安全拒绝vs临时失败 → 必须修复
  2. BF-02: 推理泄漏300字符阈值 → 建议修复
  3. TC-13: Condense摘要质量 → 建议增强（结构化模板）
  4. R3/R9: 意图分类粒度粗 → 后续优化
  5. SymbolParser: C++函数正则误判风险 → Phase 3一并解决

V4路线重排建议:

  原顺序:
    Phase 3(多Agent) → 3.5(自验证) → 4(分层上下文) → 5(Memory) → 6(事件驱动)

  建议新顺序:
    C/C++专家化 → Phase 5(Memory Bank前移) → Phase 4-Lite(结构化Condense)
    → Phase 3.5-Lite(静态检查+自检) → 规模化
    → Phase 3(多Agent,条件成熟后) → Phase 6(远期)

  重排理由:
  - Memory Bank前移: 不依赖SK升级，实现简单，用户价值极高
  - 多Agent推迟: SK升级风险高、模型能力未验证、会恶化并发
  - Phase 4/3.5拆Lite+Full: 先做不依赖外部的部分
  - 新增"规模化"步骤: V4完全没考虑80人并发
  - Phase 6保持远期: 复杂度高，当前没理由投入
