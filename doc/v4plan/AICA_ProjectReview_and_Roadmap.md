## 一、项目概况

### 1.1 产品定位

**AICA（AI Coding Assistant）** 是武汉华中数控股份有限公司的企业内部 AI 编程助手，以 VS2022 VSIX 插件形态运行，面向公司约 80 名 C/C++ 开发者提供代码生成、编辑、重构、测试、调试和代码理解服务。

**核心定位（2026-03-22 确定，产品研讨会修订）：**

> AICA 是 VS2022 内嵌的 **C/C++ 日常编码助手**，面向研发工程师（20 人）、QT 界面工程师（20 人）和算法工程师（20 人）。核心价值是**"在 VS 里随手就能用，理解你的代码和规范"**。不是通用 AI Agent，不是代码生成器，不是 OpenCode 的替代品。

**AICA 与 OpenCode 分工（研讨会决策 W2）：**

| 维度 | AICA | OpenCode |
|------|------|----------|
| 使用者 | 60 人：研发/QT/算法 | 10 人：架构/系统 |
| 使用场景 | VS 里随手用，改 1-5 个文件 | 终端里做计划性重构，改 10-50+ 个文件 |
| 交互方式 | 选中代码右键、聊天窗口 | 终端详细指令 |
| 使用频率 | 每天多次 | 每周 1-2 次 |
| 核心要求 | **快、准、不出错** | **深、广、能收尾** |

**定位理由：**

1. 公司 90% 代码是 C/C++，不需要通用能力
2. C/C++ 编码规范（Qt + MISRA C）是明确可编码的规则，LLM 可以遵循
3. CNC 领域知识（通道、轴、PLC、G-code、HNC API）是独特优势
4. 涉密环境中 AICA 是**唯一的 VS 内嵌 AI 工具**，VS 集成是不可替代的优势
5. OpenCode 已覆盖大规模重构场景，AICA 不需要重复建设

### 1.2 技术环境

| 维度 | 详情 |
|------|------|
| 技术栈 | C#/.NET，VS2022 VSIX 插件 |
| LLM | MiniMax-M2.5（220B 参数，私有部署，多模态） |
| 用户技术栈 | Qt-C++，MISRA C 标准，VS2022 |
| 开发团队 | 1 人，全程 vibe coding |
| 试用状态 | 6 人使用旧版本（f43df8a） |
| 交付节点 | 2026-03-31（3 月底） |

### 1.3 三面结构性墙壁（AICA vs Cursor/Claude Code）

| 维度 | AICA | Cursor/Claude Code | 差距本质 |
|------|------|-------------------|----------|
| 模型能力 | MiniMax-M2.5 (220B) | Claude 4.x / GPT-4o | 模型决定代码理解和生成的上限 |
| 平台 | VS2022 VSIX + IE Trident | Electron + 现代 Web | UI 渲染能力和扩展 API 丰富度 |
| 并发 | 50 并发 TTFT 207s | 云端弹性扩容 | 私有部署的物理限制 |

**核心洞察：** 对于涉密环境中的唯一 AI 工具，竞争力不来自追赶通用 AI Agent 的全面能力，而来自在特定领域的不可替代性。

> 结果：Agent 执行 48 次工具调用，撞上 50 次迭代上限，报 Agent Error，无任何有效产出。根因：① 未做任务分解 ② Edit 精确匹配频繁失败 ③ 第 25 轮后 condense 信息丢失。详见 [产品设计研讨会纪要 · 附录](../AICA_ProductDesign_Workshop.md)。

### 3.1 已完成工作中的遗留隐患

| # | 问题 | 严重度 | 所在模块 | 详情 |
|---|------|--------|----------|------|
| SEC-01 | R5 dedup 不区分安全拒绝与临时失败 | 🔴 必须修复 | ToolCallProcessor.cs | LLM 可反复尝试读取 .git/config 等受保护路径 |
| BF-02 | 推理泄漏 300 字符阈值问题 | 🟡 建议修复 | ResponseQualityFilter.cs | MiniMax-M2.5 推理文本常超 300 字符导致泄漏 |
| BF-06 | 代码解释被误分类为 Complex | 🟡 建议修复 | TaskComplexityAnalyzer.cs | 含代码片段的解释请求被错误触发规划流程 |
| TC-13 | Condense 摘要质量差 | 🟡 建议增强 | TokenBudgetManager.cs | 自由文本摘要导致长对话后 LLM 丢失上下文 |

### 3.2 产品能力短板

#### 3.2.1 无 C/C++ 专业化能力

- System Prompt 是语言无关的通用编程助手
- 三个右键命令（解释/重构/生成测试）使用通用 prompt，测试框架硬编码为 xUnit
- .aica-rules/general.md 只有通用代码质量规则
- 不了解公司 Qt-C++ 编码规范和 CNC 领域知识

#### 3.2.2 无跨文件重构能力

当前 EditFileTool 的核心问题：

| 短板 | 影响 | 对标（OpenCode） |
|------|------|------------------|
| 精确匹配失败即报错 | 模型生成的 old_string 有缩进差异时编辑直接失败 | OpenCode 有 9 层 fallback 容错策略 |
| 无重构流程引导 | 模型不知道跨文件重构应先搜全部引用再逐文件修改 | OpenCode/Claude Code 通过 prompt 引导 |
| 迭代上限固定 50 轮 | 大范围重构可能轮数不够 | OpenCode 无显式上限 |
| 无 LSP 集成 | 只能文本级搜索，不能语义级查找引用 | OpenCode 内置 clangd LSP |

**关键认知：跨文件重构的能力不在于工具或 Agent 架构，而在于模型能力。** 使用相同的 MiniMax-M2.5，AICA 和 OpenCode 的重构效果上限一致。差异仅在 Edit 容错策略和流程引导。

#### 3.2.3 无跨会话记忆

- 每次新会话都要重新解释项目背景
- 80 人团队每人每天浪费 2-3 分钟建立上下文

#### 3.2.4 无自检验证能力

- Agent 完成任务后不会自查遗漏
- 复杂分析任务覆盖率仅 50%（TC-K01）
- 长上下文后出现工具幻觉（TC-C06）

#### 3.2.5 多 Agent 协作不可行（当前阶段）

V4 文档规划的 Phase 3（多 Agent 协作）存在 4 个主要障碍：

| 障碍 | 严重程度 | 详情 |
|------|----------|------|
| 模型能力 | 🔴 关键 | MiniMax-M2.5 的 Architect 角色约束能力未经验证 |
| 并发压力 | 🔴 关键 | 多 Agent = 多并行 LLM 调用，50 并发已 TTFT 102s |
| SK 版本锁定 | 🟡 高 | SK 1.54.0 不支持 Agent Framework，升级有兼容性风险 |
| 单人开发 | 🟡 中 | 多 Agent 系统调试和维护复杂度远超单 Agent |

**决策：放弃多 Agent，在单 Agent 下通过自检步骤 + 结构化 Condense + Plan 驱动达到 80% 的效果。**

#### 3.2.6 并发规模化未规划

- V4 文档完全没有考虑 80 人并发问题
- 当前无请求队列、无优先级调度、无使用监控
- 私有部署无法弹性扩容，GPU 显存是物理上限

### 3.3 AICA 与 OpenCode 的对比分析

领导对两个工具的定位：

- **AICA**：VS 中的贴身 AI 助手，具备超强上下文感知与代码重构能力
- **OpenCode**：终端重构利器，原生终端工作流，跨文件重构能力强

两者的真实差异分析：

| 维度 | AICA | OpenCode | 差距是否真实 |
|------|------|----------|------------|
| UI 形态 | VS2022 内嵌聊天窗 | 终端 TUI | ✅ 真实差异 |
| 编辑容错 | 精确匹配 | 9 策略容错 | 🟡 可补（约 80 行代码） |
| LSP 集成 | 无（但可调用 VS SDK） | clangd 集成 | ✅ 真实差异 |
| VS 解决方案感知 | ✅ 有 | ❌ 无 | ✅ AICA 独占优势 |
| Roslyn/IntelliSense | ✅ 可调用 | ❌ 无 | ✅ AICA 独占优势 |
| 调试集成 | ✅ 可读断点/变量 | ❌ 无 | ✅ AICA 独占优势 |
| 并行子任务 | ❌ 无 | ✅ task 工具 | ✅ 真实差异 |
| 模型后端 | MiniMax-M2.5 | 同一个 MiniMax-M2.5 | ❌ 无差异（共享天花板） |
| 重构流程 | grep→read→edit 循环 | grep→read→edit 循环 | ❌ 本质相同 |
| 离线可用性 | ✅ 完全离线 | ❌ 需要配置（启动联网、LSP 下载） | ✅ AICA 优势 |

**结论（研讨会修订，决策 W2）：** AICA 和 OpenCode 明确分工——AICA 是 60 人的日常编码助手（改 1-5 个文件），OpenCode 是 10 人架构师的重构利器（改 10-50+ 个文件）。AICA 通过补齐 Edit 容错和重构流程引导覆盖日常跨文件实现需求，不追求替代 OpenCode 的大规模重构能力。

### 4.3 步骤 2：C/C++ 规范注入（Phase 1）

**目标：** AICA 生成的所有 C/C++ 代码自动遵循华中数控 Qt-C++ 编码规范。
**时间：** 1 天 | **代码量：** 0（纯规范文件）

| # | 任务 | 产出文件 | 内容 |
|---|------|----------|------|
| 2.1 | 代码风格规范 | .aica-rules/cpp-code-style.md (~400 tokens) | Allman 花括号、命名规范、m_ 前缀、Bit32 类型 |
| 2.2 | 可靠性规则 | .aica-rules/cpp-reliability.md (~350 tokens) | 内存安全、类型安全、控制流、函数规范 |
| 2.3 | 文件 I/O 规则 | .aica-rules/cpp-file-io.md (~200 tokens) | fopen/fclose 配对、落盘检查、路径规范 |
| 2.4 | Qt 专属规则 | .aica-rules/cpp-qt-specific.md (~300 tokens) | 头文件规范、成员排序、TR 翻译、.qss 颜色 |
| 2.5 | 注释规范 | .aica-rules/cpp-comment-template.md (~250 tokens) | 文件头 doxygen、函数注释格式、注释率 20-30% |

**原理：** 利用已有的 RuleLoader 管线（.aica-rules/*.md → RuleLoader → SystemPromptBuilder），规则文件通过 paths glob 仅在操作 .h/.cpp/.c 文件时激活，非 C/C++ 项目零影响。

**验收标准：** 让 AICA 生成一个新 C++ 类，检查花括号 Allman 风格、m_ 前缀、doxygen 注释、Bit32 类型是否正确。

**EditFileTool 容错匹配设计（3 层 fallback）：**

```
精确匹配（现有逻辑）
    ↓ 失败
第1层：行首缩进忽略匹配（每行 TrimStart 后比较）
    ↓ 失败
第2层：全空白规范化匹配（连续空白→单空格后比较）
    ↓ 失败
第3层：锚点搜索 + 返回最相似片段（帮助模型自修正）
```

**跨文件重构 Prompt 引导（注入 System Prompt）：**

```
当用户要求重命名/重构/批量修改时：
1. 搜索：用 grep_search 找到所有引用，记录完整文件清单
2. 计划：用 update_plan 为每个文件创建一个步骤
3. 执行：逐文件 read_file → edit → 标记步骤完成
4. 验证：用 grep_search 确认旧名称零残留
5. 完成：attempt_completion 报告所有修改文件和变更内容
```

## 八、关键决策记录

| # | 决策 | 理由 | 来源 |
|---|------|------|------|
| D1 | AICA 定位为"C/C++ 专家助手"而非"通用 AI Agent" | 公司 90% C/C++，涉密环境无竞品 | 会话 1 |
| D2 | 放弃多 Agent，采用单 Agent + 自检 + 结构化 Condense | 多 Agent 依赖 SK 升级 + 模型验证 + 并发容量，风险过高 | 会话 2 |
| D3 | Memory Bank 从 V4 Phase 5 前移 | 不依赖外部条件，实现简单，用户价值极高 | 会话 2 |
| D4 | AICA 覆盖日常跨文件实现（1-5 文件），大规模重构留给 OpenCode | AICA 和 OpenCode 明确分工，避免重复建设 | 会话 2→研讨会修订 |
| D5 | 跨文件实现通过 Edit 容错 + Prompt 引导 + 优雅中断实现 | 不需要多 Agent，单 Agent 内增强 | 会话 2 |
| D6 | 集成 GitNexus 代码智能平台（MCP 方案） | 单项 ROI 最高，Tree-sitter AST 根治正则天花板 | 会话 3 |
| D7 | 知识引擎强化降级为条件性执行 | GitNexus 成功则替代；仅在不可用时执行 | 会话 3 |
| **W1** | **AICA 定位为"VS 内嵌日常编码助手"，不追求重型 Agent 能力** | **模型天花板 + 用户高频需求集中在 F1-F3** | **研讨会** |
| **W2** | **AICA 与 OpenCode 分工：AICA 改 1-5 文件，OpenCode 改 10+** | **各自优势不同，避免重复建设** | **研讨会** |
| **W3** | **QT 工程师作为第一批重点上线用户** | **最容易出效果、拿正面反馈** | **研讨会** |
| **W4** | **F4 成功标准改为"每次有进展、多次能完成"** | **50% 一次完成率是模型天花板，需适配** | **研讨会** |
| **W5** | **路线图改为里程碑驱动（M1/M2/M3）** | **原 9 步骤对用户不可见** | **研讨会** |
| **W6** | **考核指标必须用实际任务实测** | **自评分数无法作为交付依据** | **研讨会** |
| **W7** | **每个里程碑预留 3-5 天缓冲** | **单人开发零缓冲必然延期** | **研讨会** |
| **W8** | **新增"优雅中断 + 断点续做"为 M2 核心交付** | **解决 MinCurve 案例失败体验** | **研讨会** |
