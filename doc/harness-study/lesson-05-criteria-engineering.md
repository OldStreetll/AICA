# 第五课：评估标准工程

> 日期：2026-04-05
> 来源：Anthropic "Harness Design for Long-Running Apps" + AICA v2.1 计划

---

## 核心问题

给 Evaluator "评估质量"这样的模糊指令，只会得到毫无区分度的赞美。评估标准工程 = 把"好不好"转化为具体的、可打分的、可校准的维度。

## 通用框架（4 步）

### 步骤 1：观察失败模式

让 Agent 裸跑几次，收集典型失败案例。

### 步骤 2：设计评估维度（两层）

| 层次 | 方向 | 目的 | 示例 |
|------|------|------|------|
| 防守性维度 | 从失败模式反推 | 防止模型犯常见错误 | Functionality、Consistency |
| 进攻性维度 | 从产品愿景正推 | 推动模型超越基线 | Originality、Design Quality |

Anthropic 前端设计任务的四个维度：

| 维度 | 类型 | 评什么 |
|------|------|--------|
| Design Quality | 进攻 | 配色、排版、布局的连贯视觉语言 |
| Originality | 进攻 | 自主设计决策 vs 套模板的"AI 味" |
| Craft | 防守 | 间距、对比度、排版层次——技术细活 |
| Functionality | 防守 | 能不能用 |

维度设计逻辑：每种模型常见失败模式对应一个守门员维度。

### 步骤 3：设定权重（反向补偿模型偏好）

模型总是走向阻力最小的方向。权重的本质是对模型偏好的反向补偿：

```
模型容易做到的 → 低权重（不需要额外激励）
模型天然薄弱的 → 高权重（需要额外推动）
模型不愿冒险的 → 高权重（需要额外激励）
```

### 步骤 4：迭代校准 Evaluator

```
Evaluator 打分 → 人类审查打分日志
  → 发现"发现问题但放过"的判断偏差
  → 更新 Prompt 加入针对性规则
  → 加入 few-shot 示例（"这种情况=扣分"）
  → 多轮循环直到 Evaluator 判断与人类一致
```

本质是 Prompt Engineering 的具体化：标注 → 调整 → 验证 → 迭代。

## LLM Evaluator 的两大局限

### 1. 合理化倾向（Rationalization Tendency）

LLM 倾向于为观察到的现象找合理解释而非直接判负："间距不一致，但考虑到极简风格这也可接受"。校准能改善但无法消除。

Anthropic 原文：
> "Even heavily tuned evaluators showed limits—missing subtle bugs in deeply nested features and spatial reasoning issues."

### 2. 弱模型评弱模型的可靠性问题

同一个弱模型共享同一种推理缺陷——如果 Generator 倾向于忘记同步头文件，Evaluator 也可能忽略这个问题。

**对策**：弱模型评估不应依赖开放式判断（"这段代码好不好"），而应依赖检查清单（"头文件是否同步：是/否"）。把评估从推理任务降级为模式匹配任务。

## 完整评估方案

**评估标准工程 + 人类审查 = 完整方案**

自动化评估有上限，人类审查是最后保障。不应因引入自动评估而弱化人类审查。

## AICA 映射

### C/C++ 场景的评估维度（从 MiniMax 失败模式反推）

| 维度 | 权重 | MiniMax 典型失败模式 | 类型 |
|------|------|-------------------|------|
| 一致性（Consistency） | 高 | 改了签名忘了同步头文件 | 防守 |
| 安全性（Safety） | 高 | 完成步骤但引入编译错误 | 防守 |
| 范围控制（Scope） | 中 | 做了超出范围的修改 | 防守 |
| 规范性（Conformance） | 中 | 编辑正确逻辑但破坏格式（有 M3 自动格式化兜底） | 防守 |
| 可读性（Readability） | 低 | 命名混乱、结构不清（弱模型难以评估） | 防守 |

注意：目前全是防守性维度。如果目标是"弱模型也能产出高质量代码"，需要考虑加入进攻性维度（如"方案优雅度"）。

### ReviewAgent 设计建议

- 使用**检查清单**而非开放式评估（降级为模式匹配任务，弱模型更可靠）
- 已有的用户确认流程（diff 预览 + 手动修改检测）= 人类审查层，应保留强化
- ReviewAgent 定位：辅助用户审查的 AI 第二意见，不是自动纠错器
